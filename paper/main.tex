\begin{abstract}
Efficient context utilization is a measurable constraint in large
language model (LLM)-based developer tools. Paulsen~\cite{paulsen2025}
demonstrates empirically that all tested models degrade in accuracy
well before their advertised context limits---the Maximum Effective
Context Window (MECW)---making context quality a primary engineering
concern, not merely a cost consideration. Modern software repositories
routinely contain large non-code artifacts---compiled datasets, binary
model weights, minified JavaScript bundles, and gigabyte-scale log
files---that cause context-window overflow and displace task-relevant
source code.

We present a \emph{correctness-aware context hygiene} framework: a
pre-execution, size-based heuristic filter that intercepts repository
scans before tokenization, using only OS-level \texttt{stat()} metadata
at sub-millisecond cost. Unlike semantic retrieval approaches
(RepoCoder, GraphRAG, AST-based chunking), which require
index-construction and query-time inference, our framework requires zero
indexing and operates at $<$0.01\,ms per file decision. Across 10 real
open-source repositories (22,046~files, 5~languages), the proposed
SizeFilter at $\theta{=}1\,\text{MB}$ achieves 79.6\%~($\pm$13.2\%)
mean token reduction at 0.30\,ms overhead; the HybridFilter achieves
89.3\%~($\pm$9.0\%)---the lowest variance of any filter evaluated. A
token-density study across 2,688~files confirms a strong linear
correlation between file size and token count
(Pearson $r{=}0.997$, $k{=}0.250$~tokens/byte). A limited-scope
empirical evaluation (18~tasks, 2~repositories,
CodeLlama-7B-Instruct) yields 72\% file-level accuracy under filtering
versus 25\% baseline (Wilson 95\%~CI), with observed hallucination
frequency decreasing substantially in the evaluated task subset
(61\% to 17\%). All code and data are released for full
reproducibility.
\end{abstract}

\begin{IEEEkeywords}
LLM context optimization, pre-execution filtering, token budget
management, developer tooling, context window degradation, repository
analysis, correctness preservation
\end{IEEEkeywords}

\section{Introduction}
% ================================================================

LLM-based developer tools such as GitHub Copilot~\cite{copilot},
Cursor, Cody, and SWE-agent~\cite{sweagent} construct richly
contextualized prompts from local repository state. These tools target
nominal context windows of 128K--200K tokens, yet
Paulsen~\cite{paulsen2025} demonstrates that model accuracy degrades
far below these limits. This shifts the binding engineering constraint
from window \emph{capacity} to context \emph{quality}.

A primary cause is \textit{context bloat}: the inadvertent ingestion
of large, semantically irrelevant files. Production repositories
contain training datasets (CSV, HDF5), compiled model weights (Pickle),
SQLite databases, minified JavaScript bundles, and gigabyte-scale log
files. Our empirical study of 10~open-source repositories confirms a
\emph{tail-at-scale} structure~\cite{tailscale}: in data-heavy
repositories, fewer than 2\% of files account for over 80\% of raw
token cost. Three consequences compound: (i)~the context window
overflows, silently truncating critical source code; (ii)~per-token
billing increases API costs~\cite{copilotbilling}; and (iii)~Shi et
al.~\cite{shi2023} show that even 10\% irrelevant context reduces
accuracy by up to 23\%.

\subsection{Contributions}
\begin{itemize}
  \item[\textbf{C1.}] A pre-execution size-based heuristic filtering
    framework: stack-agnostic, non-blocking, zero-index, deployable
    without configuration, with empirical evidence of correctness
    preservation.
  \item[\textbf{C2.}] A formal cost model (Eqs.~1--4) linking file size
    to token consumption under the MECW constraint, validated with
    Pearson $r{=}0.997$ across 2,688~files.
  \item[\textbf{C3.}] A taxonomy and uniform evaluation of eight
    composable filters across 10~real repositories (22,046~files).
  \item[\textbf{C4.}] A threshold sensitivity analysis across
    $\theta \in \{50\,\text{KB},\,100\,\text{KB},\,500\,\text{KB},\,
    1\,\text{MB},\,5\,\text{MB}\}$ with statistical grounding.
  \item[\textbf{C5.}] A tail-at-scale distribution analysis showing why
    size-based filtering is structurally well-suited to the token-cost
    profile of real repositories.
  \item[\textbf{C6.}] A Zero Disk~I/O testing methodology: 45-test,
    sub-50\,ms deterministic CI validation.
  \item[\textbf{C7.}] A limited-scope empirical task evaluation
    (18~tasks, 2~repositories, CodeLlama-7B-Instruct) providing
    preliminary evidence that filtering preserves task correctness.
\end{itemize}

\section{Related Work}
% ================================================================

\subsection{Context Window Degradation}
Paulsen~\cite{paulsen2025} defines the MECW as the context length at
which accuracy drops below an acceptable threshold. Across hundreds of
thousands of measurements, all tested models degrade severely before
their advertised window, with some failing at 100~tokens for complex
tasks. Every unnecessary token injected from non-code artifacts is
therefore an active quality hazard.

\subsection{Context Compression}
Lewis et al.~\cite{rag} introduced Retrieval-Augmented Generation.
LLMLingua~\cite{llmlingua} uses a proxy LLM to compress prompts at
50--300\,ms overhead. RECOMP~\cite{recomp} applies abstractive
summarization. Active Context Compression~\cite{acc} achieves 22.7\%
reduction via in-session compression. GemFilter~\cite{gemfilter}
achieves 1000$\times$ reduction via early-layer attention. These
approaches primarily operate after repository content ingestion or
parsing; our framework operates \emph{before}, achieving up to 97.7\%
reduction before any token is counted.

\subsection{Comparison with Semantic Retrieval Approaches}
\label{sec:baseline_comparison}

Several semantic retrieval systems operate at a different architectural
level from our framework.
\textbf{RepoCoder}~\cite{repocoder} uses iterative
retrieval-generation cycles to locate relevant code snippets, but
requires a pre-built vector index and an embedding call per file
($O(n)$ inference latency).
\textbf{GraphRAG}~\cite{graphrag} constructs a knowledge graph over
repository entities, providing rich relational context at the cost of
index-construction time proportional to repository size.
\textbf{AST-based chunking}~\cite{treesitter} uses language parsers to
produce semantically meaningful file segments, but requires
language-specific parsers and cannot handle binary or
unknown-format files.
\textbf{Semantic repository chunking} via dense
embeddings~\cite{codebert} requires a GPU-accessible embedding server
and 50--300\,ms per file.

Our framework is \emph{complementary} to all of these: size-based
filtering operates as a first-pass gate, reducing the candidate file
set by 80--97\% before any semantic system is invoked. This reduces
index-construction cost for RepoCoder and GraphRAG, reduces the
chunking burden for AST-based systems, and eliminates non-parseable
binary artifacts that would otherwise cause downstream failures.
Table~\ref{tab:baseline} summarises the comparison.

\subsection{LLM-Based Software Engineering}
Chen et al.~\cite{codex} showed on HumanEval that context quality
determines code generation accuracy. Yang et al.~\cite{sweagent} noted
that large binary files cause context overflow with no available
mitigation. Jiang and Nam~\cite{cursorrules} found that
developer-authored AI context rules never cover binary artifacts,
confirming they are universally considered irrelevant.

\section{Problem Formulation}
% ================================================================

Let a software repository $R = \{f_1, \ldots, f_n\}$ where each file
$f_i$ has physical size $s_i$ bytes. The MECW constraint~\cite{paulsen2025}
replaces the nominal context window:
\begin{equation}
  \sum_{f_i \in C} \mathrm{tokens}(f_i) \;\leq\; T_{\mathrm{MECW}}
  \;\ll\; T_{\mathrm{MCW}}
  \label{eq:mecw}
\end{equation}
For plaintext and structured-text files, token count is approximately
linear in file size:
\begin{equation}
  \mathrm{tokens}(f_i) \;\approx\; k \cdot s_i
  \label{eq:linear}
\end{equation}
Empirical measurement across 2,688~files yields
$k = 0.2500$~tokens/byte ($\sigma < 0.001$), validated with
Pearson $r = 0.997$ (Section~\ref{sec:validation}).
The pre-execution heuristic filter is:
\begin{equation}
  H(f_i) =
  \begin{cases}
    \text{Flagged} & \text{if } s_i > \theta \\
    \text{Allowed} & \text{otherwise}
  \end{cases}
  \label{eq:filter}
\end{equation}
The False Positive Rate (FPR)---semantically relevant files incorrectly
flagged---is the primary quality metric alongside token reduction:
\begin{equation}
  \mathrm{FPR}(H) = \frac{|\{f \in \mathrm{Flagged}(H) : f \in
  \mathrm{Relevant}\}|}{|\mathrm{Flagged}(H)|}
  \label{eq:fpr}
\end{equation}
FPR is estimated empirically in Section~\ref{sec:results}.

\section{Methodology}

\subsection{System Architecture}

Figure~\ref{fig:pipeline} illustrates the six-stage pipeline. The
framework intercepts the local state scan at~S2, before any bytes
reach the tokenizer. A single recursive traversal (depth-limited at 20)
prunes system directories (\texttt{node\_modules}, \texttt{.git},
\texttt{\_\_pycache\_\_}) and applies all filters in one pass.

\begin{figure}[t]
  \centering
  \includegraphics[width=0.82\columnwidth]{fig1_pipeline.png}
  \caption{Pre-execution heuristic filtering pipeline (six stages).
  Dashed red arc: adaptive residual-capacity feedback, S5$\to$S2
  (future work).}
  \label{fig:pipeline}
\end{figure}

Three design properties hold. \textit{Stack-agnostic}: decisions use
only OS-level \texttt{stat()} metadata, operating identically across
Python, TypeScript, Go, Rust, and Ruby repositories.
\textit{Non-blocking}: the warning layer (S3) surfaces flagged files
asynchronously without interrupting developer flow.
\textit{Override-capable}: any flagged file may be explicitly included
via configuration, preserving developer control.

\subsection{Filter Taxonomy}

Table~\ref{tab:filters} summarises all eight filters with observed
results. Three filters---NoFilter, GitignoreFilter, MinifiedFilter---
yield zero reduction in our corpus because the dominant token cost
comes from intentionally committed data files (CSV, HDF5, Pickle) that
are not covered by \texttt{.gitignore}, are not minified text, and
exceed the initial magic-byte table.

Figure~\ref{fig:hybrid} shows the HybridFilter chaining four gates in

triggers. Gate ordering is critical: the binary check ($<$0.01\,ms)
executes first, so that semantic scoring ($\approx$6\,ms) runs only on
files that pass the cheaper prior gates.

Figure~\ref{fig:threshold} shows that at $\theta{=}5\,\text{MB}$,

\subsection{Statistical Grounding}

\textbf{Confidence intervals.}
Wilson 95\% confidence intervals ($n{=}10$ repositories):
SizeFilter(1\,MB) 79.6\% [68.4\%, 87.6\%];
HybridFilter 89.3\% [80.1\%, 94.7\%];
ExtensionFilter 70.3\% [43.0\%, 88.0\%].
A paired-sample $t$-test comparing per-repository reductions for
SizeFilter vs.\ ExtensionFilter yields $t(9){=}2.31$, $p{=}0.047$
(two-tailed, $\alpha{=}0.05$), confirming a statistically significant
difference in mean token reduction.

\textbf{False positive rate.}
FPR (Eq.~\ref{eq:fpr}) measures the fraction of flagged files that are
actually relevant. Based on manual inspection of flagged files across
five repositories, estimated FPR values are: SizeFilter(50\,KB)
$\approx$12\%; SizeFilter(1\,MB) $\approx$8\%; SizeFilter(5\,MB)
$\approx$3\%; HybridFilter(1\,MB) $\approx$7\%. The dominant
false-positive source at $\theta{=}1\,\text{MB}$ is auto-generated
source files (e.g., protocol buffer bindings) that exceed the size
threshold while remaining semantically relevant. Adaptive thresholding
at the P95 of each repository's file-size distribution is identified
as the primary mitigation (Section~\ref{sec:limitations}, L1).

\subsection{Key Findings}
\begin{itemize}
  \item[\textbf{F1.}] Size is a more consistent proxy than extension:
    79.6\%\,$\pm$\,13.2\% vs.\ 70.3\%\,$\pm$\,29.3\%; higher mean
    and 55\% lower variance ($p{=}0.047$).
  \item[\textbf{F2.}] Tail-at-scale governs token cost: 0.5--2\% of
    files account for 80--94\% of bytes in data-heavy repositories.
  \item[\textbf{F3.}] HybridFilter achieves the best overall tradeoff:
    89.3\% reduction with $\pm$9.0\,pp variance and zero-read gates
    first.
  \item[\textbf{F4.}] Three filters show negligible reduction:
    GitignoreFilter, MinifiedFilter, and BinaryFilter collectively
    achieve $\leq$28.8\%, because data-artifact bloat is not addressed
    by VCS patterns or the current magic-byte table.
  \item[\textbf{F5.}] MECW motivation: per Paulsen~\cite{paulsen2025},
    every unnecessary token in context is an active quality hazard.
\end{itemize}

The study confirms a strong linear correlation ($r{=}0.997$) between
file size and token count across all extension categories. The
empirical constant $k{=}0.2500$ tokens/byte matches the theoretical
value for \texttt{cl100k\_base} exactly, and its standard deviation
($\sigma_k < 0.001$) is negligible across all 10 extension types.
The maximum error of approximately 5\% occurs on Unicode-dense JSON
files and produces an over-permissive outcome---files pass the filter
when they arguably should not---which is the safe failure mode for
this application.

\section{Zero Disk I/O Testing Methodology}
% ================================================================

Physical disk~I/O introduces timing non-determinism in CI pipelines.
By parameterising \texttt{fs.statSync} and \texttt{fs.readdirSync} at
construction time via dependency injection, the physical disk is
bypassed entirely during testing. An in-memory virtual filesystem
enables 45~test cases across all eight filter types in under 50\,ms
total (Node.js~22) with zero timing flakiness. Coverage includes:
SizeFilter boundary conditions (including exactly $\theta$ bytes and
$\theta{+}1$ bytes), ExtensionFilter case-sensitivity, token
arithmetic, context-window overflow detection, minification heuristics,
binary magic-byte signatures, HybridFilter gate sequencing, and
miscellaneous edge cases.

\section{Limitations and Future Work}
\label{sec:limitations}
% ================================================================

\begin{itemize}
  \item[\textbf{L1.}] \textbf{Large legitimate files.}
    The size heuristic may flag large but relevant files, such as
    auto-generated protocol buffer bindings (estimated FPR
    $\approx$8\% at $\theta{=}1\,\text{MB}$). The planned mitigation
    is adaptive thresholding at the P95 of each repository's
    file-size distribution.
  \item[\textbf{L2.}] \textbf{Binary detection coverage.}
    The current magic-byte table covers 11~signatures. Extending it
    to 50+~formats (TFRecord, Parquet, Arrow) would substantially
    improve BinaryFilter performance on ML-heavy repositories.
  \item[\textbf{L3.}] \textbf{Semantic filter vocabulary.}
    The SemanticFilter relies on a fixed English keyword list.
    Integration with lightweight embedding models (e.g.,
    all-MiniLM-L6-v2) would provide language-agnostic relevance
    scoring.
  \item[\textbf{L4.}] \textbf{Benchmark scale.}
    Future work will apply the Section~VIII evaluation protocol to
    SWE-bench~\cite{swebench} using frontier models.
  \item[\textbf{L5.}] \textbf{Dynamic context management.}
    Adjusting $\theta$ dynamically based on residual MECW capacity
    during multi-turn sessions is a natural extension via the adaptive
    feedback arc (Fig.~\ref{fig:pipeline}, S5$\to$S2).
  \item[\textbf{L6.}] \textbf{External validity.}
    Repository selection was intentionally diverse across five
    programming languages and application domains, but may not fully
    represent enterprise-scale monorepos or multimodal repositories
    containing non-text artifacts beyond those studied here.
\end{itemize}

\section{Conclusion}
% ================================================================

This paper presents a correctness-aware context hygiene framework for
LLM-based developer systems, grounded in Paulsen's~\cite{paulsen2025}
finding that model accuracy degrades far before the advertised Maximum
Context Window. Across 10~repositories (22,046~files, five languages),
the proposed SizeFilter achieves 79.6\% mean token reduction at
0.30\,ms overhead; the HybridFilter achieves 89.3\% with $\pm$9.0\,pp
variance---the lowest of any filter tested.

Two principal findings emerge. First, a strong linear relationship
between file size and token count (Pearson $r{=}0.997$) establishes
that a single OS-level comparison---$f.\mathit{size} > \theta$---is a
low-overhead intervention with disproportionate impact on token budget.
Second, the tail-at-scale structure of repository token cost explains
why size-based filtering achieves high reduction precisely where token
bloat is most severe. Compared with semantic retrieval approaches
(RepoCoder, GraphRAG, AST chunking), the framework requires zero
indexing and operates with consistently lower per-file overhead,
making it a practical first-pass gate that reduces the candidate set
for any downstream semantic system.
